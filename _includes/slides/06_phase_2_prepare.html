Phase 2: Prepare

Optimize the network

  • You're now dealing with at least 2 distinct networks
    • Repeat optimizations as necessary
  • If you use a VPC, figure out your cross-cloud communication strategy
    • VPN? Tunneling?
    • Consider the implications of this choice
    • SSL termination can take a good deal of RAM and CPU

Upgrade now

  • Get started on any necessary upgrades early
  • Enables more numerous and more robust migration strategies
  • Mitigates complications, complexity, and scaling
    requirements that come with upgrading while migrating
  • Similarly, "upgrade" your codebase by addressing any
    critical tech debt

So Many Metrics

  • Find metrics that give confidence that the current plan is the right plan
  • Find metrics to prevent the migration process from obscuring visibility into the overall health of the application
    • These metrics will also help prevent the migration from becoming a scapegoat for unrelated system problems
  • Find metrics that help focus the project and support the highest priorities
    • e.g. relative quantity of errors doesn't matter if errors are impacting segments that are not a priority
  • Does your monitoring infrastructure support multiple clouds?

Metrics worth considering

  • Standard application/platform health metrics
  • Actual AWS usage
  • Migration throughput
  • Number of audit successes/failures
  • Rate of phase transition (migrate -> audit -> delete)
  • Trends in delta given rate of new data vs. rate of deletion
  • How much data you think you deleted vs. how much AWS reflects
  • Recurring failures that waste compute or cause repetitive egress
  • Metrics mapping tasks to their cost-savings opportunities
