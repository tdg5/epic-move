Phase 1: Design

Keep It Simple, Stupid

  • Avoid temptation to complicate migration to solve outstanding problems
  • Prefer what is simple over what is easy
  • Minimize the number of variables whenever possible
    • Handle upgrades separately (e.g. Cassandra 1.2 -> Cassandra 2.2)
    • Handle internal service substitutions separately
    • Handle environment substitutions separately (VM -> Docker)

Beware egress costs!

  • Limit extraneous egress to save $$$
  • Running some tasks on EC2 compute might be more affordable than egress

Verifying the data

  • Decide how you're going to audit the data up front
  • Don't underestimate the confidence required to actually delete data
  • Likely a task worth running on EC2 compute
